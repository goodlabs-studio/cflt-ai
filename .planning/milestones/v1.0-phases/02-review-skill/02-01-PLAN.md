Rewrite `.claude/commands/review.md` from the current 6-step process to an 8-step process. The complete file should contain these sections in order:

**Header (keep existing):**
```
# /review — Document Evaluation Against Wiki + MCP Sources
```

**Input section (keep existing `$ARGUMENTS` reference)**

**Step 1: Parse arguments** — NEW, modeled on ask.md Step 1:
- Extract `--output` value if present: `md` (default) | `docx` | `both`
- Extract `--overlay` value if present: customer overlay name (e.g., `acme-bank`)
- Remaining non-flag tokens are file paths OR pasted content
- If any specified file path does not exist, stop with error: `Error: file not found: <path>`
- If no paths and no pasted content, stop with error: `Error: no input specified`
- Error format for unknown flag values:
  `Error: unknown --output value. Valid: md | docx | both`

**Step 1.5: Load documents** — NEW:
- For each file path from Step 1, read the file
- If pasted content (no file paths), use it directly as a single document
- Label each document by its basename for claim attribution
- Identify file type from extension (.md, .tfvars, .yaml, .txt, .tf) — treat all as text input
- Build a labeled corpus: `{ "deck.md": <content>, "vars.tfvars": <content> }`
- If `--overlay` was specified, load the customer overlay: call `resolve_stack(customer=<overlay_name>)` from `canon/stack.py` to get the active canon config. Note the overlay-specific overrides for use in Steps 2.5 and 5.

**Step 2: Extract verifiable claims** — REWRITE with structured schema (per REVW-01):
- Scan each document in corpus order (if multiple docs, process each separately)
- Extract claims using five numbered categories, in document order, section by section:
  1. Config values — specific property = value assertions (e.g., "acks=1 is recommended")
  2. Behavior assertions — X does Y under condition Z (e.g., "consumers will rebalance when...")
  3. Architecture choices — use X over Y for this use case (e.g., "prefer MirrorMaker2 for...")
  4. Metrics / limits / SLAs — numbers, thresholds, latencies (e.g., "99.99% availability")
  5. Comparisons — X is better/faster/safer than Y (e.g., "Avro is more efficient than JSON")
- Stop when all sections are exhausted. Do not invent claims not in the document.
- Output claims as a YAML block BEFORE proceeding to Step 3. This YAML intermediate is the reproducibility anchor for REVW-01:

```yaml
claims:
  - id: "<source-slug>-1"
    source_file: "<basename>"
    source_section: "<section heading or 'Introduction'>"
    category: config_value | behavior_assertion | architecture_choice | metric_sla | comparison
    text: "<exact claim text or close paraphrase>"
  - id: "<source-slug>-2"
    ...
```

- For multi-doc input: prefix claim IDs with a slug of the source filename. E.g., for `deck.md`: `deck-1`, `deck-2`. For `runbook.md`: `runbook-1`, `runbook-2`. This prevents ID collision on merge.

**Step 2.5: Challenge premises** — NEW (per REVW-02):
- Before validating individual claims, interrogate the document's unstated assumptions
- Identify 3-5 premises the document implicitly assumes (NOT explicit claims — these are deeper assumptions)
- For each premise, evaluate:
  1. Does it hold for this customer's context?
  2. If `--overlay` was specified: does the premise conflict with the customer overlay's overrides? (e.g., "under your FSI overlay, this premise about latency tolerance conflicts with market_data: sub-millisecond")
  3. Check for logical gaps: does the premise require conditions the document never establishes?
- Rate severity: Critical (blocks a key recommendation) / Moderate (weakens a claim) / Minor (pedantic)
- Output as a dedicated "## Premise Challenge" section in the report:

```markdown
## Premise Challenge

| # | Premise | Assumption | Challenge | Severity |
|---|---------|------------|-----------|----------|
| 1 | [what the document assumes] | [what must be true for this to hold] | [why it may not hold in this context] | Critical/Moderate/Minor |
```

- If no `--overlay` is set, still challenge premises against base canon and FSI overlay (if content is FSI-relevant). Specifically check FSI SLA tiers: sub-millisecond (market data), <10ms (risk), <100ms (compliance), async (reconciliation).

**Step 3: Cross-reference against wiki** — KEEP existing logic but add:
- For each claim (from YAML block), search wiki
- Note agreement, disagreement, or no wiki coverage
- Auto-stub on miss (WIKI-05): if NO wiki articles match a claim topic, append stub to `wiki/_queue.md` under `## Auto-Stubs` (same pattern as /ask)

**Step 4: Validate against MCP sources** — KEEP existing logic

**Step 5: Check canon compliance** — KEEP existing logic but add overlay awareness:
- If `--overlay` was specified, compare against the overlay-resolved canon config (not just base canon)
- Note deviations from overlay-specific overrides in the Canon Compliance table
- Add an "Overlay" column to the Canon Compliance table when overlay is active:

```markdown
| Area | Status | Overlay Override | Notes |
```

**Step 6: Generate the report** — REWRITE to include new sections and activity log emission:
- The report structure should now include these sections in order:
  1. Header (Review: <title>, Date, Source files, Scope, Claims extracted count)
  2. ## Summary (2-3 sentence executive summary)
  3. ## Claim Validation (per-section tables with Verdict column)
     - For multi-doc: organize by source file, then by section within each file
     - Each claim row includes: #, Claim, Wiki source, MCP source, Verdict (Confirmed/Corrected/Unverifiable)
     - After each section's table, list specific Corrections with explanation and source
  4. ## Premise Challenge (table from Step 2.5)
  5. ## Canon Compliance (table from Step 5)
  6. ## Gaps (unverifiable claims — candidates for further research)
  7. ## Recommendations (specific, actionable)
  8. Provenance footer: `Canon stack: <layers> | Hash: <hash> | MANIFEST: <version> | Floor: <model> | Generated: <ISO-8601>`
     - Call `provenance_footer()` from `canon/stack.py` for the canon hash part
     - Read MANIFEST.yaml version from `raw/repos/fsi-dsp/MANIFEST.yaml` (field: `version`)
     - Default floor model to the model currently answering (note it in footer)
     - Timestamp in ISO-8601 format

- Write report to `outputs/reports/<slug>-review-<YYYY-MM-DD>.md`
- If `--output docx` or `--output both`: after writing .md, invoke `python3 tools/review-to-docx.py outputs/reports/<slug>-review-<YYYY-MM-DD>.md` to generate the .docx alongside
- If `--output both`: produce both .md and .docx
- Print the file path(s) and the Summary section to the user
